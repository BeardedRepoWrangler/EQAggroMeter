# 2026-05-03

## What changed

- Researched the MacroQuest TLO surface against the `macroquest/docs` master repo. Found that three core spec assumptions don't hold in vanilla MQ: `Spawn[id].PctAggro` doesn't exist, `Raid.MainAssist` is single-valued (not `[1..3]`), and per-`raidmember` exposes no aggro % or `RaidMainAssist` flag.
- Wrote [[../decisions/0002-tlo-surface|ADR 0002]] documenting the constraint and the "honest degrade" plan for raid mode (top panel for holder/secondary/me + per-group bars only for my own raid group, with placeholders elsewhere).
- Wrote [[../design/architecture|design/architecture.md]] proposing the `lua/aggrometer/` module split: `init / data / roles / raid / ui / config / probe`.
- Built `lua/aggrometer/probe.lua` per build-order step 1 — a 2s-tick console diagnostic that dumps every documented aggro readout *plus* the unverified `Spawn[id].PctAggro` and `Raid.MainAssist[1..3]` so we can empirically confirm whether Ascendant's MQ build extends vanilla.
- Wrote [[../runbooks/run-probe|runbooks/run-probe.md]] with run instructions and the three specific output lines to look at.
- Added ADR 0002 and the architecture doc to [[../Index|Index]].

## Why

Spec said `Spawn[id].PctAggro` was the data backbone for raid-mode bars. It isn't, in vanilla. Building the UI on a phantom TLO would have produced a window where every raider outside my own group reads as 0% threat — actively misleading. Better to find out now via a 200-line probe than later via a hundred lines of UI code that lies. ADR 0002 commits us to honest degradation unless the probe shows otherwise; the probe is small enough that we can supersede ADR 0002 the same day if Ascendant turns out to ship extras.

**Confirmed:**

- `Target.AggroHolder` resolves to a real spawn (worked for the pet). ✅
- `Target.SecondaryAggroPlayer` resolves to a real spawn. ✅
- `Target.SecondaryPctAggro` semantics figured out: when I'm holder, it's the secondary's % of me (clean 0–100). When secondary == holder (an MQ quirk), it goes >100 and erratic — needs UI special-case. Documented in [[../design/architecture|architecture.md]].

**Disproven:**

- `Spawn[id].PctAggro` returns `attempt to call field 'PctAggro' (a nil value)` for every test ID. Field does not exist on Ascendant's MQ build. ADR 0002 stands — raid mode degrades as planned.

Open: still need a group probe to empirically confirm `Group.Member[n].PctAggro` moves during a fight, and a raid probe to test `Raid.MainAssist[1..3]` indexed form. Neither blocks build-order step 2 (group-mode UI), since `Group.Member[n].PctAggro` is solidly documented and the raid-mode UI design already assumes single-MA.

## Update — step 3 (role detection + filter toggles + pets) shipped

New `roles.lua` module (pure functions, no state). `data.lua` now calls `roles.tagMembers` to set `isMT`/`isMA` on each player member, then `roles.findPets` to enumerate pets via `Spawn(memberId).Pet.ID()` and append them to the roster with `isPet=true` + `ownerName` + `ownerSpawnId`. Pet aggro % derived from `Target.AggroHolder.ID == petId` (=100) or `SecondaryAggroPlayer.ID == petId` (=`SecondaryPctAggro`); 0 otherwise.

`ui.lua` now has three checkboxes (Show MT / Show MA / Show Pets, all default OFF per spec), filter logic that always shows self regardless of role tags, and the spec's pet label format `"Name (Owner's pet)"`. Single-char `[T]`/`[A]` markers appended to tagged-role names so the user can identify them when filters are on.

In-game verification: Michael ran solo, all checkboxes worked, pet appeared with correct attribution under "Show Pets", no errors. Build order step 6 (pet attribution) is also effectively done since it was inherently coupled to the "Show Pets" filter.

**Note:** Group MT/MA tagging is implemented but unverified in-game (no group available). Heuristic fallback (single tank-class member becomes MT when no explicit Group.MainTank designated) is conservative and won't auto-fire in solo.

## Update — step 7 phase 1 (config persistence)

New `config.lua` module. Per-server-per-character file at `<MQ config>/AggroMeter/AggroMeter_<server>_<character>.lua` as a hand-editable `return { ... }` table.

## Update — inter-character XTarget sharing via EQ chat channels

Michael's SK buddy can't see when DPS pulls aggro on a non-current-target mob. MQ has no TLO for "who has aggro on mob X" except the current target. Three options were on the table: heuristic loose-mobs view (no shared data), NetBots/EQBC over network (requires port forwarding/VPN), or EQ chat channel transport (in-game only).

Picked the chat-channel option because Michael (correctly) didn't want to open external ports. NetBots and EQBC are both installed on Ascendant but require network reachability between PCs. EQ's built-in `/join channelname` channels route through EQ's chat servers — no external infrastructure, no firewall changes.

New `share.lua` module. Channel name format: `agm-<leader>-XXXXX` where XXXXX is a 5-character random suffix (omits ambiguous chars 0/O/1/I) persisted in config keyed by group leader name. Re-grouping with the same leader auto-rejoins the same channel.

Wire format (compact for EQ's ~256-char chat limit):

```
AGM:Etcshadow:1234@87,5678@45,9012@100
AGM-INVITE:agm-Tankboi-3F7Q9
```

Publish at 2 Hz (under EQ's chat throttle). Receive via `mq.event` matching the standard EQ channel chat format. Remote XTargets attributed to the publishing character; expire after 6s of no refresh.

Lifecycle / cleanup per Michael's request:
- **Group channels**: persistent. 30-day TTL — sticky for regulars, forgotten for one-offs.
- **Raid channels**: ephemeral. 1-day TTL.
- TTL prune runs at script start.
- `/agm channel list` and `/agm channel forget <leader>|all` for manual control.

Slash command surface: `/agm share on|off|status`, `/agm announce`, `/agm accept`, `/agm trust on|off`, `/agm channel list|forget`. Default trust=off so invites prompt rather than auto-join.

Integration points:
- `data.lua` — replaces non-self non-pet members' `xtargets` with remote-published data when available; falls back to local heuristic attribution otherwise.
- `init.lua` — calls `share.init` after `config.init`; calls `share.tick` in the main loop; routes share-related slash subcommands.
- `config.lua` — new schema fields `share.*` and `channels.*`; bumped to version 2 (back-compat preserved via deepMerge with defaults).

Honest limitations documented in share.lua header: chat latency (~0.5–2s), rate limiting at 2 Hz, channel name length cap (~20 chars), both peers must run the script.

## Update — UI polish + attribution + cleanup before live group testing

Several iterations to clean up the per-mob sub-bar feature once Michael started using it:

- **Static player ordering** instead of sort-by-aggro — main bars now render in fixed order (MT → MA → other players), with each player's pets immediately following them. Mid-fight stability matters more than "highest threat first" for at-a-glance reading. Sub-bars within each member still sort by aggro % descending.
- **Color rules redesigned** so the bar color reflects role-correctness of the holder, not just "is this person holding":
  - Holder + holder is MT → green
  - Holder + holder is NOT MT → red (mob is in the wrong place)
  - Not holder + pct ≥ 80 → magenta (about to pull)
  - Otherwise → blue
- **Implicit MT detection in solo mode** so non-tank-class players (Michael as a necro) don't get auto-tagged as MT. New rules: solo + tank class self → self is MT; solo + non-tank-class with pet → pet is MT (necro/mage/beastlord/enchanter); otherwise no MT (color rules treat all holders as alert, accurate for "DPS solo with no pet" scenarios). Mid-iteration fix: Show MT/MA filters now exclude pets so a necro's pet-MT isn't hidden when Show MT is off.
- **Holder-attributed sub-bars improvements**:
  - Initial heuristic only attributed to pet via `Me.Pet.Target.ID` (the one mob the pet is actively swinging at). Other mobs the pet had aggro on showed under self. Generalized rule to: "if my pct < 100, attribute to MT" — covers all multi-mob fights where pet (the MT) is tanking but I have residual aggro.
  - Self's main bar pct now reflects MAX threat across all xtargets (not just current target) so the "you're about to pull" alert fires for the worst mob, not whichever you're looking at.
- **Corpse + stale slot filter**: drop XTarget entries where `Spawn.Type == 'Corpse'` or `Spawn.CleanName()` returns nil (despawned). Applied in three places (local iteration, publish payload, remote merge) so neither we nor peers get junk entries.
- **Auto-reset stale XTarget slots**: when a slot stays stale for 3+ seconds, issue `/xtarget remove <slot>` to clear it. Slot type (auto-hater) is preserved. Configurable via `xtarget.autoResetStale` and `xtarget.staleThresholdSec`. Manual immediate reset via `/agm xtreset`.

## Update — live test with SK buddy, transport pivot, UI redesign

Long debugging session getting the share working with Michael's SHD-tank buddy on Ascendant.

Discoveries and fixes:

- E3Next's bot-network filter rejected the EQ group invite at first; resolved by accepting manually before the auto-decline fired.
- `/agm announce` (which sent `AGM-INVITE:` to group chat) didn't trigger the recipient's chat hook because my mq.event pattern only matched channel chat, not group chat. Added group/raid/tell chat patterns.
- `/agm accept` extended to take an explicit channel name argument so peers could share via voice/Discord without depending on the announce path.
- Universal Chat (the EQ service that handles `/join` custom channels) is consistently down on Ascendant. Verified via manual `/join testchan` returning "Please wait until we reconnect you with the Universal Chat service. Your request has not been sent." Both peers' clients confirmed.
- **Transport pivot:** switched share from custom EQ chat channels to group/raid chat (`/g`, `/rs`). Same `AGM:`/`AGMP:` wire format, just different routing. Chat lines are now visible in group chat (filterable). Captured in [[../decisions/0003-group-chat-transport|ADR 0003]].
- Found a stale `_activeChannel` guard in `M.publish` that prevented sending after the transport switch (publish always returned early). One-line fix.
- Pet attribution bug: published payload only carried player aggro, so peer's pet held mobs were attributed to the peer player on the receiver's view. Fixed by (a) adding receive-side pet inference (any peer with `pct < 100` who has a pet → infer pet at 100%) and (b) adding `AGMP:` broadcast for the pet's swing target as a high-confidence signal.
- Held-mobs distinguishability: confirmed Spawn IDs are unique within a zone and consistent across all clients, so two mobs with the same name (e.g., two `a sepulcher skeleton`) are unambiguous in the protocol.

UI redesign (mid-session):

- Replaced per-player sub-bar layout (mobs nested under each member) with **stable mob-slot view** — XTarget-style. Each row is one mob, slot doesn't move when aggro shifts (click stability), label shows mob name + holder name + threat info.
- Switched to **two-color signal** (green = MT/pet holds correctly, red = wrong person holds → peel) per Michael's feedback that magenta + percentage was overload for the actual use case.
- Solid-color bars (fill=1.0) — color carries the entire signal, no gradient.
- Two-column text layout (mob name left, holder right) via cursor-positioning text overlay on the bar. Initial draw-list approach failed in MQ Lua's binding; cursor positioning works.
- Dropped player main bars and filter checkboxes from the UI for the pure mob-list view.

Cadence rework:

- **Event-driven publish** replacing the fixed-2s cadence. Publishes fire on holder transitions, mob add/remove, or pet target change (rate-limited to 1/sec). 15s keepalive as sanity refresh. Massively reduces idle chat noise while keeping reaction time fast.
- `/agm share on` now safe to enable while solo — broadcasts no-op when no group/raid, auto-engage when grouped. Designed for social-button use.

## Update — pet-shows-as-holder bug (Etcshadow solo)

Bug report from Michael: while soloing on Etcshadow with pet Gebanab tanking, mobs visibly switched to attacking the user (confirmed by melee damage in MQ chat) but the meter continued to show all xtargets attributed to the pet, all bars green, for several seconds before correcting. Screenshot showed 4 mobs in the meter, header `[holder: Gebanab]` on the current target, while Solodots was firing `aggro=100% (>= 90% for NORMAL) - holding rotation` — meaning `Me.PctAggro` was reading 100 on the current target.

Diagnosis: two stacking failures in the holder-attribution priority chain in `data.lua:buildXTargetsByHolder`.

1. **Priority 1 trusted `Target.AggroHolder.ID` even when `Me.PctAggro` disagreed.** `AggroHolder` lags through holder swaps. Solo necro pulling aggro from pet via DOTs hits `Me.PctAggro = 100` several ticks before `AggroHolder` updates, so the meter showed pet as holder for the duration.
2. **Priority 2 (`mob == Me.Pet.Target.ID` → pet) was a false-confidence rule.** Pet's auto-attack target ≠ pet's threat-list position. When the user out-aggros the pet on a non-current-target mob the pet is still hitting, that mob would attribute to pet via Priority 2 even though the user holds it.

Both bugs end with the meter rendering green-pet-held while the user is the one actually being hit — exact inverse of the meter's job in solo pet-class play.

Fix shipped (per [[../decisions/0004-holder-attribution-trusts-local-100pct|ADR 0004]]):

- Added Priority 0: `info.pcts[me] >= 100` on a mob → attribute to me. Local XTarget pct is ground truth; everything else is lagged or derived.
- Deleted the old Priority 2 (pet target heuristic). Heuristic + MT fallback covers the few cases it was meant to handle, and it was actively wrong otherwise.
- Renumbered the remaining heuristic block as Priority 2 (was Priority 3).
- Removed the now-unused `myPetId` and `myPetTargetId` reads from the top of `buildXTargetsByHolder`.
- Updated comment in the pet-inference block to drop the stale Priority-2 cross-reference.
- Architecture doc grew a new "Holder attribution priority" section documenting the chain.
- ADR 0004 written + linked from Index, decisions/README, and architecture.md.

Open: in-game verification still pending. If the lag was actually `Me.XTarget(slot).PctAggro` itself rather than just `AggroHolder` (architecture doc warns XTarget pct is "eventually consistent across slots"), Priority 0 might not fire fast enough on the current-target case and we'd need a UI warning state. Watching for that on next run.
