# 2026-05-03

## What changed

- Researched the MacroQuest TLO surface against the `macroquest/docs` master repo. Found that three core spec assumptions don't hold in vanilla MQ: `Spawn[id].PctAggro` doesn't exist, `Raid.MainAssist` is single-valued (not `[1..3]`), and per-`raidmember` exposes no aggro % or `RaidMainAssist` flag.
- Wrote [[../decisions/0002-tlo-surface|ADR 0002]] documenting the constraint and the "honest degrade" plan for raid mode (top panel for holder/secondary/me + per-group bars only for my own raid group, with placeholders elsewhere).
- Wrote [[../design/architecture|design/architecture.md]] proposing the `lua/aggrometer/` module split: `init / data / roles / raid / ui / config / probe`.
- Built `lua/aggrometer/probe.lua` per build-order step 1 — a 2s-tick console diagnostic that dumps every documented aggro readout *plus* the unverified `Spawn[id].PctAggro` and `Raid.MainAssist[1..3]` so we can empirically confirm whether Ascendant's MQ build extends vanilla.
- Wrote [[../runbooks/run-probe|runbooks/run-probe.md]] with run instructions and the three specific output lines to look at.
- Added ADR 0002 and the architecture doc to [[../Index|Index]].

## Why

Spec said `Spawn[id].PctAggro` was the data backbone for raid-mode bars. It isn't, in vanilla. Building the UI on a phantom TLO would have produced a window where every raider outside my own group reads as 0% threat — actively misleading. Better to find out now via a 200-line probe than later via a hundred lines of UI code that lies. ADR 0002 commits us to honest degradation unless the probe shows otherwise; the probe is small enough that we can supersede ADR 0002 the same day if Ascendant turns out to ship extras.

## Update — probe results in

Ran probe in solo on Etcshadow (necro) + Libann (pet) in Charasis. Two passes (`log/probe-output.txt`, `log/probe-output-solo-with-spawntest.txt`).

**Confirmed:**

- `Me.PctAggro` and `Target.PctAggro` are aliases that move per tick. ✅
- `Target.AggroHolder` resolves to a real spawn (worked for the pet). ✅
- `Target.SecondaryAggroPlayer` resolves to a real spawn. ✅
- `Target.SecondaryPctAggro` semantics figured out: when I'm holder, it's the secondary's % of me (clean 0–100). When secondary == holder (an MQ quirk), it goes >100 and erratic — needs UI special-case. Documented in [[../design/architecture|architecture.md]].
- `XTarget[n].PctAggro` works but is eventually consistent across slots — same mob in slot 2 and slot 5 read 55 and 63 respectively in the same tick.

**Disproven:**

- `Spawn[id].PctAggro` returns `attempt to call field 'PctAggro' (a nil value)` for every test ID. Field does not exist on Ascendant's MQ build. ADR 0002 stands — raid mode degrades as planned.

Open: still need a group probe to empirically confirm `Group.Member[n].PctAggro` moves during a fight, and a raid probe to test `Raid.MainAssist[1..3]` indexed form. Neither blocks build-order step 2 (group-mode UI), since `Group.Member[n].PctAggro` is solidly documented and the raid-mode UI design already assumes single-MA.

## Update — group-mode UI MVP (build order step 2)

Built `data.lua`, `ui.lua`, `init.lua`. Solo run on Etcshadow:

- ImGui window appeared, single bar for self, header showed `[holder: Libann]` (the pet) when target was engaged.
- Color rules: blue at 0% (correct, non-holder), magenta at ≥80% (correct after switch from yellow per request), red would fire >100%, green only when bar's spawn ID matches `Target.AggroHolder.ID`. Saw no green because the pet (not in roster yet — pets are step 6) was always the holder. Will see green in group testing when a tank in roster holds.
- Slash commands: `/agm <sub>` and `/aggro <sub>` work perfectly. `/aggrometer <sub>` does NOT — it triggers an EQ-native window (looks like a UI element auto-toggle on the client side, opens a small empty blue framed window, second invocation closes it). Three subcommands tested via `/aggrometer` (`show`, `hide`, `help`) all hit the EQ window instead of our handler. Diagnostic chat-on-entry confirmed our handler never received the command. Removed `/aggrometer` from the bind list; help text updated.

**Spec deviation logged:** original spec called for `/aggrometer` as the long form. We've dropped it. `/agm` (short) and `/aggro` (medium) cover the use case. If a future EQ patch frees the name we can revisit.

Color choice: hot magenta (RGB 0.95/0.30/0.75) for the ≥80% warning state. Subject to override in step 7 (config persistence).
