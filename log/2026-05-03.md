# 2026-05-03

## What changed

- Researched the MacroQuest TLO surface against the `macroquest/docs` master repo. Found that three core spec assumptions don't hold in vanilla MQ: `Spawn[id].PctAggro` doesn't exist, `Raid.MainAssist` is single-valued (not `[1..3]`), and per-`raidmember` exposes no aggro % or `RaidMainAssist` flag.
- Wrote [[../decisions/0002-tlo-surface|ADR 0002]] documenting the constraint and the "honest degrade" plan for raid mode (top panel for holder/secondary/me + per-group bars only for my own raid group, with placeholders elsewhere).
- Wrote [[../design/architecture|design/architecture.md]] proposing the `lua/aggrometer/` module split: `init / data / roles / raid / ui / config / probe`.
- Built `lua/aggrometer/probe.lua` per build-order step 1 — a 2s-tick console diagnostic that dumps every documented aggro readout *plus* the unverified `Spawn[id].PctAggro` and `Raid.MainAssist[1..3]` so we can empirically confirm whether Ascendant's MQ build extends vanilla.
- Wrote [[../runbooks/run-probe|runbooks/run-probe.md]] with run instructions and the three specific output lines to look at.
- Added ADR 0002 and the architecture doc to [[../Index|Index]].

## Why

Spec said `Spawn[id].PctAggro` was the data backbone for raid-mode bars. It isn't, in vanilla. Building the UI on a phantom TLO would have produced a window where every raider outside my own group reads as 0% threat — actively misleading. Better to find out now via a 200-line probe than later via a hundred lines of UI code that lies. ADR 0002 commits us to honest degradation unless the probe shows otherwise; the probe is small enough that we can supersede ADR 0002 the same day if Ascendant turns out to ship extras.

## Next

- Michael runs `/lua run aggrometer/probe` on Ascendant in solo, group, and raid contexts per [[../runbooks/run-probe|the runbook]] and shares the console output.
- Based on the probe results, either supersede ADR 0002 (Ascendant has extras → full per-raider UI is on the table) or proceed with build-order step 2: bare group-mode UI on the documented `Group.Member[n].PctAggro` surface.

## Update — probe results in

Ran probe in solo on Etcshadow (necro) + Libann (pet) in Charasis. Two passes (`log/probe-output.txt`, `log/probe-output-solo-with-spawntest.txt`).

**Confirmed:**

- `Me.PctAggro` and `Target.PctAggro` are aliases that move per tick. ✅
- `Target.AggroHolder` resolves to a real spawn (worked for the pet). ✅
- `Target.SecondaryAggroPlayer` resolves to a real spawn. ✅
- `Target.SecondaryPctAggro` semantics figured out: when I'm holder, it's the secondary's % of me (clean 0–100). When secondary == holder (an MQ quirk), it goes >100 and erratic — needs UI special-case. Documented in [[../design/architecture|architecture.md]].
- `XTarget[n].PctAggro` works but is eventually consistent across slots — same mob in slot 2 and slot 5 read 55 and 63 respectively in the same tick.

**Disproven:**

- `Spawn[id].PctAggro` returns `attempt to call field 'PctAggro' (a nil value)` for every test ID. Field does not exist on Ascendant's MQ build. ADR 0002 stands — raid mode degrades as planned.

Open: still need a group probe to empirically confirm `Group.Member[n].PctAggro` moves during a fight, and a raid probe to test `Raid.MainAssist[1..3]` indexed form. Neither blocks build-order step 2 (group-mode UI), since `Group.Member[n].PctAggro` is solidly documented and the raid-mode UI design already assumes single-MA.

## Update — group-mode UI MVP (build order step 2)

Built `data.lua`, `ui.lua`, `init.lua`. Solo run on Etcshadow:

- ImGui window appeared, single bar for self, header showed `[holder: Libann]` (the pet) when target was engaged.
- Color rules: blue at 0% (correct, non-holder), magenta at ≥80% (correct after switch from yellow per request), red would fire >100%, green only when bar's spawn ID matches `Target.AggroHolder.ID`. Saw no green because the pet (not in roster yet — pets are step 6) was always the holder. Will see green in group testing when a tank in roster holds.
- Slash commands: `/agm <sub>` and `/aggro <sub>` work perfectly. `/aggrometer <sub>` does NOT — it triggers an EQ-native window (looks like a UI element auto-toggle on the client side, opens a small empty blue framed window, second invocation closes it). Three subcommands tested via `/aggrometer` (`show`, `hide`, `help`) all hit the EQ window instead of our handler. Diagnostic chat-on-entry confirmed our handler never received the command. Removed `/aggrometer` from the bind list; help text updated.

**Spec deviation logged:** original spec called for `/aggrometer` as the long form. We've dropped it. `/agm` (short) and `/aggro` (medium) cover the use case. If a future EQ patch frees the name we can revisit.

Color choice: hot magenta (RGB 0.95/0.30/0.75) for the ≥80% warning state. Subject to override in step 7 (config persistence).

## Update — step 3 (role detection + filter toggles + pets) shipped

New `roles.lua` module (pure functions, no state). `data.lua` now calls `roles.tagMembers` to set `isMT`/`isMA` on each player member, then `roles.findPets` to enumerate pets via `Spawn(memberId).Pet.ID()` and append them to the roster with `isPet=true` + `ownerName` + `ownerSpawnId`. Pet aggro % derived from `Target.AggroHolder.ID == petId` (=100) or `SecondaryAggroPlayer.ID == petId` (=`SecondaryPctAggro`); 0 otherwise.

`ui.lua` now has three checkboxes (Show MT / Show MA / Show Pets, all default OFF per spec), filter logic that always shows self regardless of role tags, and the spec's pet label format `"Name (Owner's pet)"`. Single-char `[T]`/`[A]` markers appended to tagged-role names so the user can identify them when filters are on.

In-game verification: Michael ran solo, all checkboxes worked, pet appeared with correct attribution under "Show Pets", no errors. Build order step 6 (pet attribution) is also effectively done since it was inherently coupled to the "Show Pets" filter.

**Note:** Group MT/MA tagging is implemented but unverified in-game (no group available). Heuristic fallback (single tank-class member becomes MT when no explicit Group.MainTank designated) is conservative and won't auto-fire in solo.

## Update — step 5 (click-to-target + xtarget sub-bars) + auto-hide on EQ windows

Step 5 took several iterations:

- v1: clicks worked first time. Sub-bars rendered but were visually overpowering — full-magenta 100% bars dominated the view. Michael's screenshot showed fights with multiple mobs at 100% aggro.
- v2: changed sub-bar color rule so >=100% renders green (= "you're holding") instead of magenta (= "warning"). Added ↳ prefix and indented further. Tried `ImGui.PushStyleVar(FramePadding, 2, 1)` to compress sub-bar height — that crashed MQ's ImGui binding (signature mismatch) and corrupted the style stack so the entire window stopped rendering.
- v3: removed FramePadding override, wrapped draw body in pcall, fixed a related bug where I'd been persisting `Begin`'s `shouldDraw` return value into `_visible` instead of the close-flag — causing the window to permanently disappear after the first frame where Begin returned `shouldDraw=false`. Added `/agm reset` slash command to recover from off-screen / hidden states.

Plus: removed debug heartbeat print and the constantly-flickering "data age" footer (Michael called it out; provided no actionable info).

Also added auto-hide for EQ obstructing windows (inventory, bank, trade, merchant, bazaar, spellbook, plus Pack1..Pack10 for shift+B individual-bag windows). ImGui can't truly z-order behind EQ UI in MQ — this is the workaround. `/agm autohide on/off` toggles, `/agm windows` is a discovery command for finding additional EQ window names to add.

## Update — step 7 phase 1 (config persistence)

New `config.lua` module. Per-server-per-character file at `<MQ config>/AggroMeter/AggroMeter_<server>_<character>.lua` as a hand-editable `return { ... }` table.

Phase 1 persists: `filters` (showMT/MA/Pets), `autoHide`, `nearThreshold`, `colors` (RGBA arrays for holder/normal/near/over), `refreshMs` (per mode). Save is debounced 2s after last change to avoid disk thrash. Load happens at script start; values flow through `ui.applyConfig()` and `data.applyConfig()`. New slash commands: `/agm reload` (re-read disk), `/agm cfgpath` (print file path).

Skipped for phase 2: window position/size/opacity (need ImGui SetWindowPos + SetNextWindowSize + GetWindowPos round-trip, plus an opacity slider). Also skipped: per-character MT/MA name overrides, raid-group collapse states, Top-N count (those depend on step 4 raid mode which doesn't exist yet).

## Update — inter-character XTarget sharing via EQ chat channels

Michael's SK buddy can't see when DPS pulls aggro on a non-current-target mob. MQ has no TLO for "who has aggro on mob X" except the current target. Three options were on the table: heuristic loose-mobs view (no shared data), NetBots/EQBC over network (requires port forwarding/VPN), or EQ chat channel transport (in-game only).

Picked the chat-channel option because Michael (correctly) didn't want to open external ports. NetBots and EQBC are both installed on Ascendant but require network reachability between PCs. EQ's built-in `/join channelname` channels route through EQ's chat servers — no external infrastructure, no firewall changes.

New `share.lua` module. Channel name format: `agm-<leader>-XXXXX` where XXXXX is a 5-character random suffix (omits ambiguous chars 0/O/1/I) persisted in config keyed by group leader name. Re-grouping with the same leader auto-rejoins the same channel.

Wire format (compact for EQ's ~256-char chat limit):

```
AGM:Etcshadow:1234@87,5678@45,9012@100
AGM-INVITE:agm-Tankboi-3F7Q9
```

Publish at 2 Hz (under EQ's chat throttle). Receive via `mq.event` matching the standard EQ channel chat format. Remote XTargets attributed to the publishing character; expire after 6s of no refresh.

Lifecycle / cleanup per Michael's request:
- **Group channels**: persistent. 30-day TTL — sticky for regulars, forgotten for one-offs.
- **Raid channels**: ephemeral. 1-day TTL.
- TTL prune runs at script start.
- `/agm channel list` and `/agm channel forget <leader>|all` for manual control.

Slash command surface: `/agm share on|off|status`, `/agm announce`, `/agm accept`, `/agm trust on|off`, `/agm channel list|forget`. Default trust=off so invites prompt rather than auto-join.

Integration points:
- `data.lua` — replaces non-self non-pet members' `xtargets` with remote-published data when available; falls back to local heuristic attribution otherwise.
- `init.lua` — calls `share.init` after `config.init`; calls `share.tick` in the main loop; routes share-related slash subcommands.
- `config.lua` — new schema fields `share.*` and `channels.*`; bumped to version 2 (back-compat preserved via deepMerge with defaults).

Honest limitations documented in share.lua header: chat latency (~0.5–2s), rate limiting at 2 Hz, channel name length cap (~20 chars), both peers must run the script.
