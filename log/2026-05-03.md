## Why

Spec said `Spawn[id].PctAggro` was the data backbone for raid-mode bars. It isn't, in vanilla. Building the UI on a phantom TLO would have produced a window where every raider outside my own group reads as 0% threat — actively misleading. Better to find out now via a 200-line probe than later via a hundred lines of UI code that lies. ADR 0002 commits us to honest degradation unless the probe shows otherwise; the probe is small enough that we can supersede ADR 0002 the same day if Ascendant turns out to ship extras.

## Update — inter-character XTarget sharing via EQ chat channels

Michael's SK buddy can't see when DPS pulls aggro on a non-current-target mob. MQ has no TLO for "who has aggro on mob X" except the current target. Three options were on the table: heuristic loose-mobs view (no shared data), NetBots/EQBC over network (requires port forwarding/VPN), or EQ chat channel transport (in-game only).

Picked the chat-channel option because Michael (correctly) didn't want to open external ports. NetBots and EQBC are both installed on Ascendant but require network reachability between PCs. EQ's built-in `/join channelname` channels route through EQ's chat servers — no external infrastructure, no firewall changes.

New `share.lua` module. Channel name format: `agm-<leader>-XXXXX` where XXXXX is a 5-character random suffix (omits ambiguous chars 0/O/1/I) persisted in config keyed by group leader name. Re-grouping with the same leader auto-rejoins the same channel.

Wire format (compact for EQ's ~256-char chat limit):

```
AGM:Etcshadow:1234@87,5678@45,9012@100
AGM-INVITE:agm-Tankboi-3F7Q9
```

Publish at 2 Hz (under EQ's chat throttle). Receive via `mq.event` matching the standard EQ channel chat format. Remote XTargets attributed to the publishing character; expire after 6s of no refresh.

Lifecycle / cleanup per Michael's request:
- **Group channels**: persistent. 30-day TTL — sticky for regulars, forgotten for one-offs.
- **Raid channels**: ephemeral. 1-day TTL.
- TTL prune runs at script start.
- `/agm channel list` and `/agm channel forget <leader>|all` for manual control.

Slash command surface: `/agm share on|off|status`, `/agm announce`, `/agm accept`, `/agm trust on|off`, `/agm channel list|forget`. Default trust=off so invites prompt rather than auto-join.

Integration points:
- `data.lua` — replaces non-self non-pet members' `xtargets` with remote-published data when available; falls back to local heuristic attribution otherwise.
- `init.lua` — calls `share.init` after `config.init`; calls `share.tick` in the main loop; routes share-related slash subcommands.
- `config.lua` — new schema fields `share.*` and `channels.*`; bumped to version 2 (back-compat preserved via deepMerge with defaults).

Honest limitations documented in share.lua header: chat latency (~0.5–2s), rate limiting at 2 Hz, channel name length cap (~20 chars), both peers must run the script.

## Update — pet-shows-as-holder bug (Etcshadow solo)

Bug report from Michael: while soloing on Etcshadow with pet Gebanab tanking, mobs visibly switched to attacking the user (confirmed by melee damage in MQ chat) but the meter continued to show all xtargets attributed to the pet, all bars green, for several seconds before correcting. Screenshot showed 4 mobs in the meter, header `[holder: Gebanab]` on the current target, while Solodots was firing `aggro=100% (>= 90% for NORMAL) - holding rotation` — meaning `Me.PctAggro` was reading 100 on the current target.

Diagnosis: two stacking failures in the holder-attribution priority chain in `data.lua:buildXTargetsByHolder`.

1. **Priority 1 trusted `Target.AggroHolder.ID` even when `Me.PctAggro` disagreed.** `AggroHolder` lags through holder swaps. Solo necro pulling aggro from pet via DOTs hits `Me.PctAggro = 100` several ticks before `AggroHolder` updates, so the meter showed pet as holder for the duration.
2. **Priority 2 (`mob == Me.Pet.Target.ID` → pet) was a false-confidence rule.** Pet's auto-attack target ≠ pet's threat-list position. When the user out-aggros the pet on a non-current-target mob the pet is still hitting, that mob would attribute to pet via Priority 2 even though the user holds it.

Both bugs end with the meter rendering green-pet-held while the user is the one actually being hit — exact inverse of the meter's job in solo pet-class play.

Fix shipped (per [[../decisions/0004-holder-attribution-trusts-local-100pct|ADR 0004]]):

- Added Priority 0: `info.pcts[me] >= 100` on a mob → attribute to me. Local XTarget pct is ground truth; everything else is lagged or derived.
- Deleted the old Priority 2 (pet target heuristic). Heuristic + MT fallback covers the few cases it was meant to handle, and it was actively wrong otherwise.
- Renumbered the remaining heuristic block as Priority 2 (was Priority 3).
- Removed the now-unused `myPetId` and `myPetTargetId` reads from the top of `buildXTargetsByHolder`.
- Updated comment in the pet-inference block to drop the stale Priority-2 cross-reference.
- Architecture doc grew a new "Holder attribution priority" section documenting the chain.
- ADR 0004 written + linked from Index, decisions/README, and architecture.md.

Open: in-game verification still pending. If the lag was actually `Me.XTarget(slot).PctAggro` itself rather than just `AggroHolder` (architecture doc warns XTarget pct is "eventually consistent across slots"), Priority 0 might not fire fast enough on the current-target case and we'd need a UI warning state. Watching for that on next run.

## Update — combat tap recursion crash + fix

`/agm combat tap on` crashed EQ on first attempt. Root cause was a self-feedback loop in the tap output:

- `mq.event` patterns match phrases like `#1# YOU for #2# points of damage.` and `#1# tries to #2# YOU#*#`.
- The tap handler echoed the verbatim triggering line back to chat as a debug field — meaning the echo line itself contained `YOU for X points of damage.` and matched our own pattern.
- That re-fired the handler, which echoed again with the previous echo line nested inside, which fired again, line length growing each iteration.
- Either chat throttle, recursive call depth, or sheer message volume crashed the EQ client.

Lesson worth keeping: any debug output from inside an `mq.event` handler must NOT contain the trigger pattern. The captured `#1#` placeholder content is safe (it lies between the pattern's literal delimiters, not inside them); the raw `line` argument is never safe to re-emit verbatim.

Fix:

- `combat.lua:dispatch` reordered to resolve+mark FIRST, then handle tap. Resolution is the actual job; tap is a debug aid.
- Tap output now echoes only the captured attacker prefix (`attacker='a sepulcher skeleton hits'`) plus source and match count. The raw `line` is no longer echoed.
- Added a 250ms rate limit on tap echoes as defense-in-depth — even if a future pattern's capture were to accidentally re-match, the rate limit caps amplification at 4 echoes/sec. Suppressed echoes are counted and surfaced via `/agm combat status` (`tap-dropped=N`) so we can see when we're under-sampling.
- `combat.setTap(false)` resets the drop counter.

The crash teaches one more thing: tap is a fragile debug feature inside an event-firing context. The probe-style file-log approach (write events to a file, view after) might be a more robust pattern for future similar tools. Not changing this one now since the rate-limit + capture-only approach should be sufficient.

```
[E3] DAMAGE! for 119 damage from A pyre golem hits
[AggroMeter] combat detection: ttl=5.0s tap=true recent attackers=1 tap-dropped=1
[AggroMeter] mob 356 (a pyre golem): 2.0s ago
[AggroMeter:combat] miss | attacker='A pyre golem' matches=0    ← bug
[AggroMeter:combat] hit  | attacker='A pyre golem hits' matches=1
[AggroMeter:combat] miss | attacker='A pyre golem' matches=0    ← bug
[AggroMeter] combat tap = false
```

What worked:

- Hit pattern fires on real EQ damage lines on this server. ✓
- Attacker resolves against the xtarget index, mob 356 attributed to me. ✓
- `/agm combat status` reflects the recent-attacker correctly with a fresh timestamp. ✓
- Rate limit kicks in benignly (`tap-dropped=1`), no recursion, no crash. ✓
- `combat tap = false` cleanly exits.

What didn't:

Miss pattern fires (so we know `#1# tries to #2# YOU#*#` matches real lines), but `matches=0` — the captured attacker text didn't resolve to any xtarget mob. Cause: `#1#` for the hit pattern captures `<mob> <verb>` (e.g., `A pyre golem hits`) while `#1#` for the miss pattern captures only `<mob>` (e.g., `A pyre golem`) because the verb is in `#2#` of the miss pattern. The resolver had a single `normalizeAttackerLine` that *always* stripped the trailing word as a verb, which corrupted the miss case (`A pyre golem` → `A pyre`).

Fix:

- Replaced the single-shot normalizer with `attackerCandidates(s)` which returns up to three candidates in priority order: as-is (handles miss), verb-stripped (handles hit), possessive-stripped (handles limb-attack hit).
- `resolveAttacker` walks the candidate list and uses the first one that lands on a real xtarget index entry. This handles both pattern shapes without needing the resolver to know which pattern fired.
- Comment near `onHit` updated to reflect the new resolver behavior.

Bonus observation from the chat log: the EQ damage line starts with capital `A pyre golem` (sentence-start capitalization) but the xtarget `CleanName()` returns lowercase `a pyre golem`. Both pass through `normalizeMobName` which lowercases — so the case difference is already handled. Confirmed working in the hit path; will be confirmed working in the miss path on the next tap run after this fix deploys.

## Update — tap output rerouted from /echo to a log file

Michael flagged that the `/echo`-based tap was unreadable in chat — too much volume, scrolls past everything else, hard to find the lines you're looking for. Switched tap output to file-based logging modeled on `probe.lua`:

- Output path: `<MQ Logs dir>/aggrometer-combat.log`. Falls back to `lua/aggrometer/` then `.` if the Logs dir TLO can't resolve.
- Append mode, line-buffered (`setvbuf('line')`), so a crash doesn't lose the tail.
- Per-event line format:
  ```
  YYYY-MM-DD HH:MM:SS  hit   attacker="A pyre golem hits"  matches=1  mobIds=[356]
  YYYY-MM-DD HH:MM:SS  miss  attacker="A pyre golem"        matches=1  mobIds=[356]
  ```
- `tap on` writes a `---- tap session start: <timestamp> ----` banner; `tap off` writes a session-end summary with event count.
- `combat.setTap(true|false)` now returns `(success, info)` — `info` is the log path on tap-on, the event count on tap-off, or an error message if the file couldn't be opened. Slash command surfaces these.
- Removed the chat rate-limit code path (`_tapRateMs`, `_tapDropped`) — file writes don't have the same recursion / chat-throttle concerns that motivated it. All output goes to the file at full volume.
- `/agm combat status` now prints the tap log path and current event count when tap is on.
- `/agm combat tap on` echoes the path to chat plus a hint: `(tail this file in PowerShell: Get-Content -Wait <path>)` so Michael can stream it live in another window if he wants.

Side benefit: file output kills the chat-recursion crash vector outright. Earlier versions guarded against it by avoiding the trigger phrase in the echo plus a rate limit; with no chat output at all there's nothing for our patterns to re-match against.

## Update — combat-event broadcast (ADR 0006) — wire protocol shipped

After local detection verified clean (hits and misses both resolving to xtarget mobs in the tap log, no recursion crash, no false negatives) Michael asked to wire up the peer-broadcast extension. Shipped in this session:

**Wire format added** — `AGMH:<charName>:<mobId>,<mobId>,...`. Full-snapshot semantics (replace, not delta). Empty publisher set → no broadcast (let TTL handle cleanup). See [[../decisions/0006-combat-event-broadcast|ADR 0006]] and the updated [[../design/wire-protocol]] for the spec, and the cadence section for how it integrates with the existing event-driven publish loop.

**`combat.lua` refactor** —

- `_attackedMe` (single map) split into `_localAttackers` (self, per-event TTL 5s) + `_remoteAttackers[charName]` (peers, per-publish TTL 30s).
- New public APIs: `recentAttackerCharOf(mobId)`, `localAttackerSet()`, `ingestRemoteAttackers(charName, mobIds)`, `setRemoteTtl(s)`, `remoteTtl()`, `myCharName()`. Backward-compat shim retained for `recentAttackerOf(mobId)` (= "did *self* attack").
- `init(charName)` accepts the local character name so peer self-echo can be filtered by both share.lua dispatch and combat.lua ingest (defense in depth).
- `gc()` walks both maps with their respective TTLs and prunes empty per-peer tables.

**`share.lua` additions** —

- New `buildAttackerPublishPayload()` reads `combat.localAttackerSet()` and emits `AGMH:<me>:<sortedMobIds>` (sorted for deterministic output, helps log diffs / regression tests). Returns nil on empty set.
- `M.publish()` now sends AGM, AGMP, and AGMH each gated on their own emptiness check — peers see the full state every cycle but no empty-payload spam.
- `M.tick()` change-detection also tracks an `attackerSetKey()` membership hash (sorted mobIds joined) and triggers a publish when the set adds or removes a mob. Per-event timestamp churn within an unchanged set is ignored — keepalive carries the freshness instead.
- New `handleAGMHData(sender, body)` parses the line, defensively re-filters self-echo (already filtered by dispatch but cheap to layer), and calls `combat.ingestRemoteAttackers`.
- `dispatchAGM` prefix branches reordered: `AGMH:` (5 chars) and `AGMP:` (5 chars) checked before `AGM:` (4 chars). Lua substring matching makes the order incidental here (the colon makes each prefix unique) but longest-first is a clearer pattern.
- `M.debug()` now dumps all three current sample payloads — useful for "is the wire format empty for the right reasons?" troubleshooting.

**`data.lua` Priority -1 generalization** — was `if combat.recentAttackerOf(mobId) then holderId = mySpawnId`; now resolves the attacker char (self or peer) and looks up their roster spawn ID. Falls through cleanly when the char isn't in the local roster (e.g., peer is in your group but you somehow don't have them in members yet).

**`config.lua` defaults** — `combat.remoteAttackerTtlSec = 30.0` added alongside the existing `combat.attackerTtlSec = 5.0`. Doc comment explicitly notes the keepaliveMs*2 lower bound.

**`init.lua` slash command surface** — `/agm combat status` updated to surface remote peers separately from local attackers, with explicit local/peer labels in the per-mob lines:

```
combat detection: ttl=5.0s  remote-ttl=30.0s  tap=true
  local recent attackers: 2   peers heard: 1 (3 entries)
  local mob 433 (a pyre golem): 1.4s ago
  peer Tankboi mob 599 (a giant rat): 4.2s ago
```

**Edge cases reasoned through in ADR 0006:**

- Peer not running AggroMeter — no AGMH published, attribution falls through to existing priorities. No regression.
- Same mob with two same-named instances — combat.lua already over-attributes via the matches-list fallback; each peer publishes the over-attributed set; receiver applies to all. Consistent.
- Pet hits ON the player still match `YOU for ...`; pet hits ON the pet do NOT match (no "YOU"). AGMH propagates "peer is being hit"; "peer's pet is being hit" still relies on the existing pet-inference rule. Acceptable.

## Update — TTL-asymmetry race observed in live test, fixed via empty-AGMH-on-transition

Michael ran the AGMH wire-protocol broadcast in a real group fight with his SK buddy. Result: **AGMH path is working** — buddy's bar correctly attributed the mob to Etcshadow via received AGMH — but a screen-vs-screen disagreement surfaced.

Captured state from the live test:

- **Buddy's screen:** header `[holder: Vtik]` (from `Target.AggroHolder.CleanName()`, raw TLO), bar red on `Etcshadow` (from priority chain, AGMH receive)
- **Michael's screen:** bar green on `Vtik` (his pet, the actual current holder)
- **Ground truth at moment of screenshot:** mob was on Vtik; Michael was standing safely behind, not being hit

The buddy's bar was wrong — they thought the mob was on Etcshadow, but it had retargeted to the pet some seconds earlier. Walked through the timeline with Michael:

1. Mob ramps Michael for one swing — combat events fire on Etcshadow, `_localAttackers[mobId]=now`, AGMH publishes `AGMH:Etcshadow:<mobId>` to group chat.
2. Buddy receives, stores `_remoteAttackers[Etcshadow][mobId]=now`.
3. Mob retargets to Vtik. Vtik takes the next several swings. Etcshadow stops taking hits.
4. After 5s with no fresh events, **Michael's** `_localAttackers[mobId]` ages out (local TTL default 5s).
5. Michael's priority chain falls through to Priority 1 (`Target.AggroHolder=Vtik`), bar correctly shows green-on-Vtik.
6. **Buddy's** `_remoteAttackers[Etcshadow][mobId]` is still well within the 30s remote TTL. Their bar stays red-on-Etcshadow until that TTL expires — up to 25 more seconds of staleness.

Root cause: TTL asymmetry. Local TTL is short (5s, refreshed per event) so Etcshadow correctly forgets quickly. Remote TTL is long (30s, must exceed share.keepaliveMs*2 so a single dropped chat message doesn't age out a peer prematurely). The two TTLs serve different purposes and have to be different — but the gap shows up as cross-peer disagreement on the visible UI.

**Fix shipped:** when the local attacker set transitions from non-empty to empty, publish `AGMH:<me>:` (empty body) once. Receivers parse the empty body as "publisher has no current attackers" and immediately clear the entry. Cross-peer staleness window drops from up to 30s to ~1 publish cycle (~1s). Pure-downtime case (set stays empty) is unaffected — change-detect in `tick()` doesn't fire, no broadcasts go out.

Implementation:

- `share.lua:buildAttackerPublishPayload(forceEmpty)` accepts an optional flag; returns `AGMH:<me>:` (empty body) when set.
- `share.lua:M.publish(forceEmptyAttackers)` accepts the flag and forwards it.
- `share.lua:tick()` computes `justEmptied = (currentAttKey == '' and _lastAttackerKey ~= '')` and passes that into publish.
- `share.lua:handleAGMHData` already supported empty-body parsing (`combat.ingestRemoteAttackers` clears the entry when given empty mob list) so no receiver-side changes were needed.

ADR 0006 updated to refine the "empty set: don't broadcast" rule into the more precise "empty set during downtime: don't broadcast / empty set on transition: broadcast once." wire-protocol.md updated to spec the empty-body semantics. Considered but skipped: applying the priority chain to the header text (`[holder: ...]`). Would have made this case actively worse — buddy's header was showing the correct `Vtik` from raw TLO; if it had used the priority chain it would have shown the stale `Etcshadow` instead.
